Privacy & Security

How Ravi protects your code and data

Our Core Privacy Promise

Ravi is built with privacy at its core. We access only what's truly required to deliver intelligent code reviews and process everything transiently. Your sensitive data is never stored or retained after analysis.

What Ravi Accesses

Database Metadata

Ravi queries database metadata to understand schema structure:

  • Table and column names - Schema structure only
  • Data types and constraints - Column definitions, foreign keys, indexes
  • Table statistics - Estimated row counts, table sizes, index usage
  • Query performance metrics - Statistics from pg_stat_* views

Code and Pull Requests

Ravi analyzes code to provide accurate reviews:

  • Repository code - To analyze code changes in context
  • Pull request metadata - Title, description, author, etc.
  • Code changes - Diffs and modified files in pull requests

Code Is Never Persisted

Your code is processed transiently during analysis and is immediately discarded. We fetch code from GitHub, analyze it for the review, and then delete it. Your source code is never stored in our databases or retained after the review is complete.

This transient processing model is standard practice across modern AI-powered developer tools.

What Ravi Never Accesses

Sensitive Database Content

  • Actual data rows - We cannot execute SELECT queries on your application tables
  • Customer PII - Names, emails, addresses, phone numbers in your database
  • Sensitive business data - Orders, transactions, payments, user records
  • Passwords or credentials - Any secrets stored in your database tables

Environment Secrets

  • Environment variables - .env files, configuration secrets
  • API keys and tokens - Third-party service credentials in your code
  • Private keys - SSH keys, SSL certificates, signing keys

How Database Access Works

When you configure database access for Ravi, you create a special read-only PostgreSQL user with strictly limited permissions. This user can only:

  • Read information_schema and pg_catalog (metadata)
  • Read pg_stat_* views (statistics)
  • Execute system functions like pg_table_size()

The read-only user cannot:

  • Execute SELECT queries on your application tables
  • Run write operations (INSERT, UPDATE, DELETE)
  • Modify schema (CREATE, ALTER, DROP)
  • Manage database users or roles

Defense in Depth

Creating a read-only user with limited permissions is an additional security precaution and follows database hardening best practices. Ravi's code is designed to only query metadata; we do not read actual customer data from your tables, regardless of what permissions are granted. The restricted user provides an extra layer of protection and ensures that even in the unlikely event of a compromise, your sensitive data remains inaccessible.

Your Control

You have full control over what Ravi accesses:

  • Revoke database access anytime by removing the configuration in your dashboard
  • Uninstall the GitHub App to stop receiving PR analysis
  • Delete your account to remove all stored credentials and metadata
  • Rotate database passwords as often as you like (update in dashboard after rotation)
  • Restrict network access by IP address using your firewall rules

Security Best Practices

We recommend:

  • Use strong passwords (16+ characters, alphanumeric + symbols)
  • Enable SSL/TLS for database connections with sslmode=verify-full
  • Restrict network access to your database by IP address
  • Use a read replica if possible to isolate analytics workloads
  • Rotate passwords regularly (every 90 days recommended)
